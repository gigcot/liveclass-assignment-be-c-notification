# LAB-003: 비동기 발송 처리 — Transactional Outbox + Relay + Worker

## 구조 요약

```
접수 API ─[트랜잭션]→ DB (UserNotification + OutboxEvent)
                              ↓
                    Relay (여러 인스턴스 가능)
             FOR UPDATE SKIP LOCKED, scheduled_at <= NOW()
                              ↓
                        Redis Queue
                              ↓
                    Worker (여러 인스턴스 가능)
                      발송 Mock → 상태 업데이트
```

## 접수 흐름

1. `templateId` 없으면 `notificationType`으로 최신 템플릿 조회
2. 템플릿 + `referenceData`로 메시지 렌더링 (플레이스홀더 치환 완료)
3. `[트랜잭션]` UserNotification + OutboxEvent 저장 — `NotificationRegistrationPort`가 담당, `@Transactional`은 어댑터에 위치

## OutboxEvent 구조

```
OutboxEvent:
  id
  notification_id  - UserNotification FK
  status           - PENDING / RELAYED
  scheduled_at
  created_at

payload (JSON):
  userId
  channel          - EMAIL / IN_APP
  renderedMessage  - 렌더링 완료된 메시지 (Worker는 바로 발송만)
```

## Success Criteria

- 알림 접수 시 UserNotification과 OutboxEvent가 하나의 트랜잭션으로 저장된다
- templateId 없이 접수해도 notificationType 기준 최신 템플릿으로 렌더링된다
- Relay 여러 인스턴스를 동시에 띄워도 동일 OutboxEvent가 중복으로 Redis에 적재되지 않는다
- Worker가 Redis에서 꺼낸 알림을 발송(Mock) 후 SendStatus를 SENT로 업데이트한다
- 발송 실패 시 RetryInfo가 업데이트되고, 3회 초과 시 FAILED로 전이된다
- 예약 발송(scheduledAt)은 별도 스케줄러 없이 Relay 폴링 조건으로 처리된다
